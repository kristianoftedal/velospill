Product

What Velospill Does

A full-featured fantasy sports platform tailored to competitive cycling — from live drafts to tactical orders.

🏁

Live Draft System

Real-time snake draft with Pusher presence channels. Participants see who's online, live picks appear instantly, and an auto-pick timer via QStash handles AFK managers.

Real-time
🏆

Scoring & Standings

Dynamic point scoring from race results with JSONB config tables — rule changes never require code deployment. Standings computed on-demand with race result aggregation.

Data-driven
🔄

Transfer Market

Waiver wire system with priority by standings. Managers bid for dropped riders, admin approves, and transfers execute within configurable windows. Ownership-at-race-time scoring preserved.

Strategic
⚔️

Strategic Orders

12 order types including boosts, sabotages, and counters. Orders like Domestique, Breakaway, Hammer, and Shimanobil add a tactical layer with risk/reward mechanics.

Tactical
🏥

IR (Injury Reserve)

Managers can request IR status for injured riders — admin approves, freeing roster slots while preserving team history. Status flows: pending → approved/rejected → return eligible → returned.

Roster Mgmt
🗓️

Race Calendar & Leagues

Admin assigns races to leagues. Each league scopes to its assigned race calendar — no bleed-through between leagues. Supports classic races, stage races, and grand tours.

Development Workflow

Built with GSD

Get Shit Done — an AI-driven development workflow using Claude Code agents for planning, execution, and verification.

What is GSD?

GSD is a structured workflow for human-AI collaborative software development. The developer provides intent and decisions; Claude Code agents handle planning, writing code, committing atomically, and verifying results. Each unit of work flows through a dedicated agent pipeline — planner → executor → verifier — with the human reviewing and approving at each gate.

01

Milestone Planning

Human defines the next milestone goal. GSD agents research the codebase, generate a phase roadmap, and surface architectural decisions for human sign-off.

02

Phase Research

Before each phase, a research agent explores the codebase and external docs to produce a RESEARCH.md that informs the planner.

03

Plan Generation

A planner agent converts the phase goal into an executable PLAN.md with discrete tasks, file targets, verification criteria, and done-conditions.

04

Plan Checking

A checker agent reviews the plan for goal coverage, scope sanity, and must-have derivation. Issues loop back to the planner (max 2 iterations).

05

Atomic Execution

An executor agent implements each task, committing atomically per task. Human reviews diffs. Deviations are logged and surfaced immediately.

06

Verification

A verifier agent does goal-backward analysis: did the codebase actually deliver what the phase promised? VERIFICATION.md classifies as passed / human-needed / gaps.

07

Quick Tasks

Ad-hoc fixes that don't need a full phase: /gsd:quick spawns planner + executor for atomic, tracked mini-tasks with STATE.md updates.

08

State Tracking

STATE.md, ROADMAP.md, and per-phase directories persist full context — any session can resume from exactly where it left off with zero context loss.

Architecture

Key Engineering Decisions

28 recorded decisions — each one captured at the time it was made, with the phase and rationale preserved.

JSONB Scoring Config

Scoring rules stored as JSONB in the database, not hardcoded. Rule changes (Norwegian cycling federation updates mid-season) deploy without code changes.

Phase 1

Ownership-at-Race-Time

Points are awarded to the team that owned the rider when the race was ridden. Transfers don't retroactively steal points — historical fairness guaranteed.

Phase 6

Three-Query Assembly

Avoids SQL JSON_AGG complexity by running three targeted queries and assembling results in application code. Reusable pattern across riders, teams, and standings.

Phases 16–19

roster_slots as Source of Truth

A dedicated table for current roster composition. All write paths (draft, drop, transfer, IR) atomically update roster_slots in the same transaction — no derived counts.

Phase 23

Pusher Presence Channels

Real-time draft without a persistent WebSocket server. Pusher presence channels provide participant tracking compatible with Vercel's serverless runtime.

Phase 4

COALESCE Stage Roll-up

Grand tour stages grouped to parent race using COALESCE(parentRaceId, id) — no schema changes needed, consistent with the existing race data model.
